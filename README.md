# info-align

> 模糊任务 → 结构化需求确认单。一次问完，不再猜。

## 一句话

用户说"帮我做个客户流失预测"——信息缺口至少有数据来源、技术栈、输出形式、评估指标。与其让 AI 逐个追问或直接盲猜，**info-align 把缺失维度打包成一次交互**，用户点完就交付下游执行。

## 效果

```
用户输入                              输出
──────────────────────────          ──────────────────────────
"帮我做一个客户流失预测模型"   ──►   ✅ 已确认：客户流失预测模型
                                    ❓ 数据来源 → CSV | 数据库 | API
                                    ❓ 技术栈 → sklearn | XGBoost | PyTorch
                                    ⚠️ 评估指标 → AUC+F1 (默认)
                                    ⚠️ 数据失衡 → SMOTE (默认，⚠大数据量OOM)
                                               ↓ 用户点选/确认
                                               ↓
                                    下游 Skill 直接消费 ResolvedFields[]
```

整个过程用户只交互一次——看到一份带默认值的确认单，改不对的，确认对的。

## 触发规则

**默认对所有任务触发**，除非同时满足：每个动作无歧义、无隐性维度需决策、无可选格式/风格/工具/技术选型。

严格豁免（下面这些不触发）：

- 纯事实查询（"Python 是什么"）
- 读文件 / 展示内容，不涉及修改或生成
- 用户给了非常具体的步骤级指令

## 流水线

```
请求 → [Phase 1] 收集上下文 → [Phase 2] 显式提取 → [Phase 2.5] 四象限探测
                                         │
                     用户点选确认 ← [Phase 4] 生成确认单 ← [Phase 3] 合并去重+状态判定
                                         │
                                      [Phase 5] 解析结果 → [Phase 6] 交付下游
```

每个阶段的输出 schema 就是下一阶段的输入 schema。用户只看到 Phase 4 产出的确认单，内部 JSON、象限标签、status 枚举全部不可见。

## 两个交互分支

Phase 4 根据平台能力自动选择：

| 分支 | 条件 | 交互方式 |
|------|------|---------|
| A | 平台提供原生问答工具（如 Claude Code 的 `AskUserQuestion`） | 可点击按钮，一次展示最多 7 个字段 |
| B | 无原生问答工具 | 纯文本逐字段交互（A/B/C 选项 + 自定义输入） |

无论哪个分支，`missing` 字段优先于 `defaulted` 字段，推荐选项标注 "(推荐)"，默认值标注 "(默认)"。

## 特殊场景

| 用户说 | 行为 |
|--------|------|
| "随便"/"你决定" | 全取推荐值，跳过交互，告警后交付 |
| "直接做，不用确认" | 跳过对齐流程，全默认 |
| 极度模糊（如"帮我分析一下"） | 象限❸ 触发维度展开，先帮用户聚焦再收集需求 |
